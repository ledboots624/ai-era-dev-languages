言語別評価マトリクス

PythonはAI開発適性◎・エコシステム成熟度◎でPriority 0（即時採用）、信頼度high。NumPy/PyTorch/FastAPIからなる成熟したエコシステムが強み。TypeScriptはAI開発適性○・エコシステム成熟度○でPriority 1（並行導入）、信頼度medium。Vercel AI SDK・tRPCが実用的なAI統合を支援。RustはAI開発適性△・エコシステム成熟度△（candle・burnは未成熟）でPriority 2（条件付き検討）、信頼度low〜medium。GoはAI開発における評価不能（エビデンス不足のため比較対象外）。

推奨アーキテクチャ：Python＋TypeScript二層構造（信頼度: high）

動的型付けのPythonがAIデファクトである一方、プロダクション化では型安全性が重要になるという「型安全性の逆説」が、この二層アーキテクチャの構造的根拠である。

【探索フェーズ】      【API境界】                   【プロダクションフェーズ】
Python           →   OpenAPI / JSON Schema    →   TypeScript
（動的型・柔軟性）     （FastAPI + Pydantic v2）      （静的型・安全性）

FastAPI + Pydantic v2がこの境界を架橋する「接着剤」として機能する。この組み合わせの習得は最もROIの高い技術投資とされる。

フレームワーク選定指針：抽象度のトレードオフ（信頼度: medium）

高抽象度（LangChain）はPoC・初期検証向けで迅速なプロトタイピングが可能だが、破壊的変更への脆弱性・デバッグ困難リスクあり。中抽象度（LlamaIndex・Pydantic AI）はRAG・構造化出力向けで用途特化の明確さが利点だが汎用性に制限あり。低抽象度（Instructor・直接API）は本番・高信頼性要件向けで完全な制御・透明性が得られるがボイラープレートが増加する。

最適戦略：プロトタイプは高抽象度で素早く検証し、本番化時に中〜低抽象度に移行する。移行コストを最小化するためにアダプターパターンで業務ロジックをフレームワークから隔離する設計が必須（信頼度: medium〜high）。

マルチプロバイダー対応（信頼度: medium）

本調査はOpenAI APIを事実上のデフォルトとして参照しており、Anthropic Claude・Google Gemini・Mistral等の比較は不十分である。実務指針として、provider: strパラメータを設計初日から組み込み、Provider Adapterパターンにより各プロバイダーを等価に扱える構造にすること。マルチプロバイダーSDK比較は重大情報ギャップ（G1）として追加調査が必須。
